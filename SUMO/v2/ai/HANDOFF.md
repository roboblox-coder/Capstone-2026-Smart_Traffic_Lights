## Known risks — check these before assuming a bug is in the new code

1. **`traci.start(..., label=...)` may not exist on very old SUMO
   builds.** `sumo_env.py` and the probe code both use labelled
   connections so the probe and the training session don't share one
   global TraCI. If you get an error like *unexpected keyword 'label'*,
   the user needs to upgrade SUMO/traci — *do not* strip the labels,
   they're load-bearing.
2. **TLS id might not match the user's network.** The default
   `3153556582` is the right one for `NE_8th_St_Corridor.net.xml`. If
   the user has regenerated the network, list the available IDs:
   ```bash
   python -c "import traci, sumolib; traci.start([sumolib.checkBinary('sumo'), '-c','sim.sumocfg','--start','--quit-on-end']); print(traci.trafficlight.getIDList()); traci.close()"
   ```
   …then pass `--tls-id ...` to the trainer and update `TLS_ID` in
   `run_websocket_ai.py` (lines near the top, in the CONFIG block).
3. **State-size / action-size drift.** If the user retrains on a
   different intersection, `run_websocket_ai.py` will detect the
   mismatch at load time and refuse to drive the lights — it falls
   back to actuated and the frontend shows
   `AI: fallback:state_size_mismatch`. Re-train against the same TLS.
4. **`differential` reward can look bad early on.** Episode-1 reward
   may be near zero or negative. The signal is in the *trend* over
   episodes. Don't change the reward function in the first run; train
   for at least 20 episodes before judging.
5. **Frontend WebSocket URL is hardcoded** to `ws://localhost:8765`.
   If you change `WS_PORT` in `run_websocket_ai.py`, also update
   `connectWebSocket()` in `frontend/index.html` (around line 1230).
6. **`run_websocket_ai.py` calls `env._total_waiting_time`** —
   wait, it doesn't. It defines its own `total_waiting()` inside main.
   But `eval.py` and `train_dqn_sumo.py` do reach into
   `env._total_waiting_time()` for logging only. That's a deliberate
   shortcut; don't refactor it on the way to Wednesday.
